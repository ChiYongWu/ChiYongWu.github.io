4. 打断点

 可以对单一请求进行拦截，也可以对所有请求进行拦截。具体看你在哪打断点。

 选择你要拦截的请求路径，鼠标右键勾选断点：

 打完断点后，重新请求页面。当页面请求该断点时，会自动弹窗：

注意事项：

  • 触发断点后，既会拦截请求，也会拦截响应，仔细看清楚。
  • 打断点的位置有讲究，最好打准位置，省得拦截到不必要的请求/响应路径。
  • 拦截篡改并放行后，若不需要再拦截，可以禁用拦截。
  • 打断点后，清空拦截到的请求，刷新页面也会重新拦截到。

6. 映射本地资源

 有的APP放行拦截的请求后，页面提示网络超时，请重新尝试。这是因为APP客户端判定服务器响应超时，所以客户端放弃等待。此时，若我们要做UI测试，比如将标题字段增加到255个，观察是否会有穿模。可以考虑用map local功能。

6.1 map local

 功能：将匹配的请求返回对应的响应结果，不经过原目标服务器。
 应用场景：用于测试环境不改动的情况下，本地调试，一般是静态资源更新，走本地资源而不需要重新构建测试环境。

6.2 map local操作流程

 step1：拦截请求后，右键鼠标选择“save response”保存响应信息到json文件，然后对json文件的内容进行修改。

 step2：复制请求的URL

 菜单栏 -> Tools -> map local

 设置好后，下次再访问该请求时，不会请求服务器。而是直接返回本地的静态资源。

 应用场景：测试时，当你不想要重新构建环境时，可以用这种方法替换文件，走一遍流程。看bug是否成功修复。

7. 巧用map remote切换测试环境

7.1 map remote

 功能：将请求的目标服务器替换为指定的服务器。
 应用场景：客户端不改变的情况下，切换后台测试环境。

8. 接口调试

 若要进行接口测试，比如需要对请求进行重放，可以使用repeat功能重放。若要模拟请求发送，可以对需要模拟的请求选择compose，然后修改请求参数后execute。
